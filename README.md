# qa-toolkit

> Browser extensions and Claude Code plugins for **QA, IT, and BA** workflows
> on the web. Click any element on a live site → log a styled issue with
> screenshot, computed styles, console + network logs, and a Figma deeplink.
> Sync to Jira via Claude Code. More extensions coming.

> **Renamed from `qa-annotator-extension` on 2026-05-10**. GitHub redirects
> the old URL automatically — existing clones, plugin installs, and bookmarks
> keep working without action. Update your git remote at leisure:
> `git remote set-url origin https://github.com/dongquoctien/qa-toolkit.git`

## What's in this monorepo

```
qa-toolkit/                                 ← this repo
├── extensions/
│   └── qa-annotator/        ← v0.6.0 — bug reporter + annotator (shipped)
├── shared/                  ← reusable code (placeholder until 2nd extension)
├── plugins/
│   └── qa-tooling/          ← Claude Code plugin (4 slash commands)
├── docs/                    ← cross-extension docs + design archive
└── .claude-plugin/
    └── marketplace.json     ← plugin marketplace metadata
```

Two pieces, one repo. They share two JSON contracts:

- **`qa-profile.json`** — project conventions (URL patterns, framework,
  locales, sections, Jira / Figma defaults). Generated by the plugin,
  consumed by every extension.
- **`qa-figma-tree.json`** — cached Figma frame tree. Lets extensions
  resolve a picked DOM element to its nearest Figma frame.

| Piece | What it does | Audience |
|---|---|---|
| **[qa-annotator](extensions/qa-annotator/README.md)** | Click any element → log a QA issue. Multi-pick, auto-cropped screenshot, canvas annotation editor (pins / rect / arrow / blur / text), 6 QA modes, console + network capture, axe-core a11y scan, Figma auto-fill. | QA engineers |
| **[qa-tooling plugin](plugins/qa-tooling/README.md)** | `/qa:init` auto-detects stack → `qa-profile.json`. `/qa:figma-sync` crawls Figma. `/qa:sync` pushes reports to Jira via MCP. `/qa:doctor` health-check. | Dev / QA leads |

## Quick install

### Extension (qa-annotator)

**For QA team (no git):** download the latest ZIP from
[Releases](https://github.com/dongquoctien/qa-toolkit/releases/latest),
unzip, then in Chrome:

1. `chrome://extensions` → enable **Developer mode**
2. **Load unpacked** → select the unzipped folder
3. Pin the **QA Annotator** icon to the toolbar

**For devs (clone + load source):**

1. `git clone https://github.com/dongquoctien/qa-toolkit.git`
2. `chrome://extensions` → **Developer mode** → **Load unpacked**
3. Select **`extensions/qa-annotator/`** (NOT the repo root — Chrome needs the
   folder containing `manifest.json`)

Click the icon → **Settings** → import a `qa-profile.json` from any project,
optionally import its `qa-figma-tree.json`. Without imports, the extension
uses a bundled empty profile and works on any URL.

Detailed setup: [`extensions/qa-annotator/docs/INSTALL.md`](extensions/qa-annotator/docs/INSTALL.md).

### Plugin (qa-tooling)

In a Claude Code session at any project:

```
/plugin marketplace add dongquoctien/qa-toolkit
/plugin install qa-tooling@qa-annotator
```

Then run `/qa:init` to generate `docs/qa/qa-profile.json` for that project.
Update via `/plugin marketplace update qa-annotator`.

Full plugin docs: [`plugins/qa-tooling/README.md`](plugins/qa-tooling/README.md).

## Daily workflow

```
1. QA opens target site → starts Inspector → clicks elements
2. Each click: modal opens with auto-captured cropped screenshot
                    + auto-filled Figma deeplink (when tree imported)
                    + per-element computed styles
3. QA fills Title / Severity / Expected → Save
4. End of session: popup → Export ZIP
5. Dev runs /qa:sync at the project → Jira sub-tasks created
```

| 5 | **qa-visual-regression** | 🟢 backlog | Snapshot all pages from `qa-profile.pages[]` × viewport → diff vs baseline → flag pixel diff. Uses `chrome.downloads` for storage (avoids `chrome.storage.local` quota). |
| 6 | **ba-requirement-capturer** | 🟢 backlog | Click element → "describe this requirement" → export user-story Markdown → push to Jira description. |
| 7 | **qa-jira-quick-action** | 🟢 backlog | Toolbar popup: log time, change status, assign without opening Jira. Auto-detects ticket from branch name. |

## Why a monorepo

- **Shared artifacts**: every extension consumes `qa-profile.json` /
  `qa-figma-tree.json`. Single source of truth for the schema.
- **Shared UI**: form modal, focus trap, severity pill, sidebar TOC are
  generic enough to reuse. `shared/` will host them once a second extension
  needs them.
- **Single Claude Code plugin**: `/qa:init`, `/qa:sync` etc. serve every
  extension. No duplication.
- **AI-friendly**: Claude Code sees full context in one repo, suggests
  consistent patterns across extensions.
- **Per-extension publishing**: each `extensions/<name>/` is a complete MV3
  manifest. `Load unpacked` on the folder, package via
  `extensions/<name>/scripts/package-extension.mjs`, submit to Chrome Web
  Store independently. Repo structure invisible to the end user.

---

## Repo conventions

- **No build step** for any extension. Plain JS / IIFE attaching to
  `self.QA.<module>`. Manifest content_scripts list controls load order.
- **Versioning**: each extension versions independently. Tag pattern:
  `qa-annotator@v0.6.0`, `qa-i18n-coverage@v0.1.0`. Plugin version tracked
  in `plugins/qa-tooling/plugin.json` AND `.claude-plugin/marketplace.json`
  (must bump both — see CLAUDE.md gotcha).
- **Permissions discipline**: each extension declares only what it needs in
  its own `manifest.json`. No "swiss army knife" permissions on qa-annotator.
- **Storage budget**: `chrome.storage.local` quota = 5–10 MB total. Big
  artifacts (visual baselines, video) go to `chrome.downloads` or IndexedDB.

## Contributing

1. New extension? Open an issue first describing the problem and target
   workflow. Extensions need to share at least one of `qa-profile.json`,
   `qa-figma-tree.json`, or `shared/` modules — otherwise consider a
   standalone repo.
2. Bug fix on an existing extension? PR against `main`, scoped to one
   `extensions/<name>/` folder if possible. Run the affected extension's
   smoke test (load unpacked, exercise the changed flow).
3. Touching `shared/`? Verify every extension that imports the changed
   module still loads. Bump the affected extensions' minor version.
